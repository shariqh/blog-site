The honest answer is that I don't always know. Some weeks I ship fast and feel alive. Other weeks I'm grinding through a problem that should take an hour and is taking three days, and I have to remind myself what drew me here in the first place.

But there are patterns. Things that show up across projects, across years, across the gap between the code I wrote when I was learning and the code I write now.

**Building something that didn't exist before.**

This sounds obvious, but I mean it literally. The moment a thing I made starts doing what I imagined — a component that snaps together right, a pipeline that transforms the data the way I sketched it — something clicks. I've tried to find that feeling in other kinds of work and I haven't.

It's not about the output. I could reach a similar end result by delegating or using a no-code tool. The feeling is specific to the making. To the path from nothing to something.

**Code is one of the few mediums that fights back honestly.**

Writing fights back too, but it lets you convince yourself. You can look at a sentence and think it's good when it isn't. Code doesn't allow that. It either runs or it doesn't. It either handles the edge case or it blows up in production. There's a moral clarity in that I find useful.

I work with a lot of ambiguity: product decisions, architectural tradeoffs, team dynamics, the shifting definition of "done." Code is where I go when I need reality to be a bit more legible. A test passes or it fails. The cursor moves to the right column or it doesn't.

**The compounding.**

Early in my career I thought experience in this field meant knowing more APIs. Now I think it means having better instincts for when to trust a fast solution and when to slow down. That's not something I could have told myself — I had to write a lot of code I later had to throw away.

The compounding isn't the code itself. It's the calibration. Each project adds a thin layer of context to every future decision. I notice when a pattern feels right because it's been right before, and I notice when something feels wrong for reasons I can't fully articulate yet. That gap between the feeling and the articulation is where most of the real learning lives.

**Occasionally, because it surprises me.**

I've been doing this long enough that most of what I write is familiar. The structure of an API handler, the shape of a hook, the way errors need to propagate. But sometimes a problem is genuinely new, strange enough that I have to actually think. And I still like that. More than I expected after this many years.

Recently that's been AI-adjacent work: wiring together agent loops, understanding where models fail, building tooling that helps me think faster rather than just automate the thinking. It's new enough that I'm a beginner again in places, which is uncomfortable and also good.

---

None of this is a manifesto. People come to this work from many directions and stay for many reasons, and most of them are legitimate.

But I've noticed that when people ask "why do you still write code yourself?" — especially as someone who manages engineers and spends a lot of time in strategy and systems — the implied answer is: you probably shouldn't anymore.

That's not where I've landed. Code keeps me connected to the material reality of what we build. It gives me something to do with my hands when thinking in abstractions gets too frictionless. And it keeps some part of me in a beginner's relationship with the craft, which I think makes me a better person to work alongside.

The answer shifts a bit every year. Holding onto the fact that there is an answer seems like the part worth protecting.
