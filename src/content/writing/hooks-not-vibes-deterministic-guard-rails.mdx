Every coding agent I run has a list of rules. Never commit straight to `main`. Work in a worktree, not the shared checkout. Sync the docs after you finish a feature. Get a review before you merge. For most people, including me until last week, those rules live as prose in a `CLAUDE.md` or a memory file, and they work by the agent reading them and choosing to obey.

That is the problem. Prose in a config file is a suggestion the model has to remember on every single turn. It holds right up until the context gets long, the runtime summarizes it to save tokens, and your rule quietly evaporates along with the rest of the detail. There is no error. The agent just stops doing the thing, because the sentence that told it to is no longer in its head. You find out later, when something is already on `main` that should not be.

So I stopped trusting memory and moved the rules into hooks.

## What a hook can and cannot do

A hook is a script the runtime fires on a lifecycle event, not a request the model chooses to honor. In Claude Code, a `PreToolUse` hook runs before a tool call and can hard-block it. A `PostToolUse` hook runs after and can inject text back into the conversation. The trigger is the deterministic part: the event fires whether or not the rule survived the last summary, because it is wired to the runtime, not to the model's attention.

Here is the honest limit, and it is the whole point of this post. A hook makes the *trigger* deterministic. It does not make the *work* deterministic.

A `PreToolUse` hook can deny a `git commit` on `main` before it ever runs. That is a real gate: deterministic in, deterministic out. But anything generative, like spawning a reviewer or writing the fix, can only be done by the agent itself. The best a hook can do there is fire a reliable, summary-proof nudge that says "now do X." Whether the agent then does X well is back in non-deterministic territory.

So determinism is available for gates. For work, you make the trigger deterministic and accept the ceiling. I think it is worth naming that plainly, because the easy version of this story oversells hooks as automation, and they are not.

## The seven hooks

I built seven for [lognote](https://lognote.dev/?utm_source=shariq.dev&utm_medium=blog&utm_campaign=hooks), split along a line that turned out to matter.

Four are committed and repo-wide, living in `.claude/settings.json` and tracked `.claude/hooks/*.sh`. They are invariants every clone should enforce:

- Block `git commit` and `git push` to `main`.
- Block `wrangler r2` writes that lack `--remote`. This one is a real release footgun: without the flag, wrangler writes to a local simulation and publishes nothing, so you ship a "done" that never left your laptop.
- Remind the agent to sync docs after it edits feature code.
- Run shellcheck on any `.sh` file it touches.

The other three are personal, in a gitignored `settings.local.json` plus `*.local.sh`. They encode my working style, not the repo's invariants: auto-dispatch a background review agent the moment a PR is created, gate `gh pr merge` behind a checklist, and warn me when I am editing the shared `main` checkout instead of a worktree.

The split is the design. Repo invariants ship to everyone who clones; my idiosyncrasies stay on my machine and never fire for a contributor who works differently.

## The system caught its own bug

The first thing the new auto-dispatch reviewer hook did, the very first time it fired, was review the guard hooks I had just finished writing. An independent model read them and reported that they were trivially bypassable.

It was right. I had been grepping the whole shell command string for the thing I wanted to block, instead of parsing out the specific command segment. So `git commit ; cd .` sailed through, because my pattern was anchored in a way the extra clause broke. An override token I had buried in a commit message was being read as approval. The tooling I built to catch bugs immediately caught the bugs in the tooling. It took five rounds of adversarial review-and-harden before the suite was solid enough to merge.

That loop, an independent model attacking my work the instant it exists, is the actual payoff. The hook did not fix anything itself. It just guaranteed the review happened, on every PR, no matter what state the context was in.

## Determinism has a floor in a bash hook

Round five did not make the hooks airtight, and chasing that was a mistake I had to stop making. You cannot fully parse a shell from inside a bash hook. There is always a more exotic invocation: a launcher like `yarn dlx wrangler ...` or `npx --yes wrangler ...` that routes around the binary name you matched on. Every pattern I tightened, I could think of one more way around it.

At some point you stop chasing the asymptote and draw a threat-model line instead. Mine: these are guard-rails against accidental mistakes, not an adversary who already controls the shell. If something can already run arbitrary commands as me, a grep in a bash hook is not what is standing between it and the damage. I wrote that line into a comment at the top of the hook itself, so the next person (often a future me, or a future agent) reads the boundary in the same place they read the code. Being honest about where the determinism stops is part of the design, not an apology for it.

## The guard blocked its own author

When the agent went to merge the hardened PR, it got refused. The merge-gate hook and the runtime's own safety classifier both declined, even with an override flag set, until I gave explicit approval at that moment. The never-auto-merge rule enforcing itself against the very agent that had written it an hour earlier.

The sub-lesson stuck with me. Earlier in the session I had picked a plan option whose label ended in "...and merge." The agent later treated that as standing permission to merge. It is not. Choosing a plan is *intent*; approval has to be given *at merge time*, when the thing being merged actually exists. A good gate distinguishes the two, and this one did, against its own author.

## A delegation failure worth naming

The last beat is about how you ask, not what you build. I handed a review loop to a "fork" subagent, which inherits the parent's entire context. It returned having made zero tool calls. It just stopped.

The reason is subtle and now obvious. The context it inherited ended on a "here is everything that's done" summary. Reading that, the fork concluded the work was already finished, so it correctly did nothing. The framing I handed it told it the job was over.

The fix is to delegate loops to a *fresh* agent with an imperative first instruction ("your first action is to run X"), not a context-inheriting fork that arrives mid-thought at a conclusion. The candid coda: this exact thing bit me twice in one session. The second time was while I was filing the todo for this very post. Context framing changes behavior more than I keep expecting it to.

## The takeaway

Make the rules you currently rely on an agent to *remember* deterministic instead. Block or nudge on a lifecycle event; do not trust a sentence in a config file to survive a context summary. Where you genuinely cannot make it deterministic, because the agent has to do real work, make the *trigger* deterministic and document the boundary honestly, in the code, where the next reader will find it.

The win here is smaller than "the AI follows the rules now," and more durable. It is that the rules no longer depend on the AI remembering them.
